# Observatory Deployer Guide

A 3-minute walkthrough for demonstrating the stack after cloning.

## 1. Prerequisites

* Docker
* Docker Compose
* Terraform (for Grafana provisioning)
* 2 GB free RAM

## 2. Start the stack

Initialize the environment and start the containers:

```bash
cp .env.example .env
make up
```

Wait approximately 30 seconds for all service health checks to pass.

## 3. Provision Grafana

Apply Terraform to create datasources, dashboards, folders, and alert rules:

```bash
cd terraform && terraform apply -auto-approve && cd ..
```

This takes ~5 seconds. Grafana is now fully configured with 3 datasources, 3 dashboards, and 2 alert rules.

## 3. Generate traffic

Populate the system with data using one of these methods:

* **Quick:** Runs `make validate` to health-check all services (Alloy, Gateway).

```bash
make validate
```

* **Deep validation:** Runs `scripts/validate_trace_log_correlation.py` to synthetically inject W3C traceparent contexts and run 10 correlation checks in ~5 seconds.

```bash
python scripts/validate_trace_log_correlation.py
```

* **Sustained:** Runs `tools/load-generator.py` to generate realistic traffic with rate limiting and error injection, populating dashboards with live data.

```bash
python tools/load-generator.py --rate 10 --duration 120
```

* **SLO demo:** Runs `tools/fault-injector.py` to deliberately generate failure traffic against `/orders` (which returns 503 because the downstream service is absent). This drives the SLO burn-rate alerting rules and makes the burn-rate dashboard panels non-empty.

```bash
# Verify the endpoint returns 503 first
python tools/fault-injector.py --probe-only

# Run for 5 minutes to see stable burn-rate data
python tools/fault-injector.py --duration 300 --rate 10 --error-ratio 0.3
```

## 4. Explore in order

Open http://localhost:3000 (admin/admin) and present the dashboards in this exact sequence:

1. **Service Health RED dashboard** (`grafana/provisioning/dashboards/service-health-red.json`, UID: `service-health-red`): Point out the 6 panels showing request rate, error rate, latency percentiles, service status, and volume.

2. **SLO Burn Rate dashboard** (`grafana/provisioning/dashboards/slo-burn-rate.json`, UID: `slo-burn-rate`): Point out the burn rate stat panel, which will show an active value calculating error budget burn.

3. **System Overview** (`grafana/provisioning/dashboards/system-overview.json`, UID: `system-overview`): Point out the Scraped Services count.

4. **Trace Detail:** Click any trace in the Tempo panel to show the trace detail view.

5. **Correlated Logs:** Click "Logs" in the trace view to show Loki logs instantly correlated via traceID.

## 5. What to tell the interviewer

* "Every panel queries real data — no mock responses, no sample datasets"
* "The SLO rules are generated by Sloth, not hand-written PromQL"
* "Trace-to-log correlation uses OTLP/HTTP structured metadata, not regex extraction"
